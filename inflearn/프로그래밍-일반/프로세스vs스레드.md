# 🖥️ 프로세스(Process)와 스레드(Thread)의 차이

사용자가 제공한 비유("프로세스는 집이고 스레드는 그 집에 사는 사람")를 바탕으로, 프로세스와 스레드의 개념 및 차이점을 직관적이고 깔끔하게 정리한 문서입니다.

---

## 🏠 비유로 이해하기

| 구성 요소 | 비유 | 설명 |
| :--- | :--- | :--- |
| **프로세스 (Process)** | **집 (House)** | 독립된 공간을 가지며, 다른 집의 공간을 함부로 침범할 수 없습니다. |
| **스레드 (Thread)** | **가족 구성원 (Person)** | 한 집(프로세스) 안에 살며, 거실과 주방(자원)을 함께 공유합니다. |

* **독립된 공간 (프로세스):** 각 집은 주소(메모리 공간)가 다르고 서로 분리되어 있습니다.
* **자원 공유 (스레드):** 같은 집에 사는 사람들은 냉장고(데이터)나 거실(힙 메모리)을 공유하여 효율적으로 생활합니다.

---

## 🔍 핵심 개념 및 차이점 비교

### 1. 프로세스 (Process)
* **정의:** 운영체제로부터 시스템 자원을 할당받는 **작업의 단위**입니다. (실행 중인 프로그램)
* **특징:**
    * 각 프로세스는 독립된 메모리 공간(주소 공간)(Code, Data, Stack, Heap)을 가집니다.
    * 기본적으로 다른 프로세스의 자원에 접근할 수 없으며, 접근하려면 IPC(Inter-Process Communication)라는 복잡한 통신 과정이 필요합니다.
    * 하나의 프로세스가 컴퓨터를 다운시켜도 다른 프로세스에는 영향을 주지 않습니다. (집 한 채가 무너져도 옆집은 안전한 것과 같습니다.)

### 2. 스레드 (Thread)
* **정의:** 프로세스가 할당받은 자원을 이용하는 **실행의 흐름 단위**입니다.
* **특징:**
    * 프로세스 내에서 각각 Stack만 따로 할당받고, **Code, Data, Heap 영역은 공유**합니다.
    * 주방과 거실을 공유하듯, 스레드 간 데이터 교환이 매우 빠르고 효율적입니다.
    * **주의점:** 같은 공간을 공유하기 때문에 한 스레드가 문제를 일으키면(예: 주방에 불을 냄) 프로세스 전체(집 전체)가 영향을 받을 수 있습니다.

---

## 📊 한눈에 보는 비교표

| 비교 항목 | 프로세스 (Process) | 스레드 (Thread) |
| :--- | :--- | :--- |
| **개념** | 실행 중인 프로그램 (독립적) | 프로세스 내의 실행 흐름 (의존적) |
| **메모리 공유** | 공유하지 않음 (독립된 영역 보유) | Stack 외에 Code, Data, Heap 영역 공유 |
| **생성/소멸 비용** | 무겁고 많은 비용 발생 | 가볍고 빠른 생성/소멸 |
| **통신 비용 (IPC)** | 높음 (공동 작업이 까다로움) | 낮음 (자원을 공유하므로 빠름) |
| **장애 영향** | 다른 프로세스에 영향 없음 | 하나의 스레드 장애가 전체 스레드 종료로 이어질 수 있음 |

---

## 💡 요약
* **프로세스**는 완전히 독립된 프로그램 실행 단위이며, **스레드**는 그 프로그램 안에서 효율적으로 일 처리를 나누어 하는 일꾼들입니다.
* 멀티 태스킹을 위해 무작정 프로세스를 늘리는 것보다, 자원을 공유하여 가볍게 동작하는 멀티 스레드를 활용하는 것이 현대 소프트웨어 아키텍처에서 매우 중요합니다.


# 쓰레드(Thread)가 프로세스(Process)보다 빠른 이유: 문맥 교환(Context Switching) 비용 중심 분석

시스템 프로그래밍과 운영체제 아키텍처에서 **"왜 쓰레드가 프로세스보다 더 빠른가?"**에 대한 해답은 핵심적으로 **문맥 교환(Context Switching)의 비용 차이**와 **메모리 공유 모델**에 있습니다.

---

## 1. 핵심 요약 (TL;DR)

| 비교 항목 | 프로세스(Process) 문맥 교환 | 쓰레드(Thread) 문맥 교환 |
| :--- | :--- | :--- |
| **공유 메모리** | 없음 (독립된 주소 공간) | 있음 (상위 프로세스의 메모리 공유) |
| **캐시 플러시** | **필요** (TLB 및 CPU 캐시 초기화) | **불필요** (동일 주소 공간 유지) |
| **가상 메모리 복사** | MMU 레지스터(CR3 등) 전환 필요 | 전환 불필요 (동일한 페이지 테이블 사용) |
| **오버헤드 수준** | 매우 높음 (Heavyweight) | 매우 낮음 (Lightweight) |

---

## 2. 프로세스와 쓰레드의 메모리 구조 차이

쓰레드가 문맥 교환 시 왜 비용이 적게 드는지 이해하려면, 먼저 두 개체의 메모리 구조적 차이를 알아야 합니다.

* **프로세스(Process):** 운영체제로부터 독립된 메모리 공간(Code, Data, Heap, Stack)을 할당받습니다. 하나의 프로세스는 다른 프로세스의 메모리에 직접 접근할 수 없습니다.
* **쓰레드(Thread):** 하나의 프로세스 내부에서 생성되며, 프로세스의 **Code, Data, Heap 영역을 공유**합니다. 오직 자신만의 독립적인 **Stack 영역과 레지스터 세트**만 개별적으로 가집니다.

---

## 3. 문맥 교환(Context Switching) 비용이 차이 나는 이유

문맥 교환이란 CPU가 한 작업에서 다른 작업으로 전환될 때, 현재 작업의 상태(Context)를 저장하고 다음 작업의 상태를 로드하는 과정을 말합니다.

### ① 가상 메모리 주소 공간(Address Space)의 전환 유무
* **프로세스 교환:** CPU는 완전히 새로운 주소 공간으로 이동해야 합니다. 이를 위해 MMU(메모리 관리 장치)의 페이지 테이블 루트 포인터 레지스터(예: x86의 `CR3` 레지스터)를 새 프로세스의 페이지 테이블 주소로 변경해야 합니다. 이 작업은 비용이 큽니다.
* **쓰레드 교환:** 같은 프로세스 내의 쓰레드 간 전환 시에는 주소 공간이 동일하므로 페이지 테이블을 교환할 필요가 없습니다. 레지스터와 스택 포인터 등 최소한의 CPU 상태만 변경하면 됩니다.

### ② TLB (Translation Lookaside Buffer) 캐시 오버헤드 (핵심 원인)
* **프로세스 교환:** 주소 공간이 바뀌면 기존 프로세스가 사용하던 가상 주소-물리 주소 변환 정보가 완전히 무효화됩니다. 따라서 하드웨어 캐시인 **TLB가 플러시(Flush, 초기화)**됩니다. 
    * *결과:* 문맥 교환 직후 새로운 프로세스가 실행될 때 대규모 **TLB 미스(Miss)**가 발생하여, 느린 메인 메모리에 수차례 접근해야 하므로 극심한 성능 저하가 일어납니다.
* **쓰레드 교환:** 주소 공간이 동일하므로 **TLB를 플러시할 필요가 없습니다.** 이전 쓰레드가 로드해 둔 메모리 변환 주소를 그대로 활용할 수 있어 캐시 적중률이 유지됩니다.

### ③ CPU 데이터 캐시(L1, L2, L3) 효율성
* 프로세스가 바뀌면 CPU 내부의 데이터 및 명령어 캐시(Cache)에 저장되어 있던 데이터들이 무효화되거나 대치(Replace)되어 **캐시 미스(Cache Miss)**가 자주 발생합니다.
* 쓰레드 간 교환 시에는 동일한 데이터 영역(Heap, Data)을 공유하며 작업하는 경우가 많기 때문에 CPU 캐시에 데이터가 남아있을 확률(캐시 지역성, Locality)이 훨씬 높습니다.

---

## 4. 결론

쓰레드가 프로세스보다 빠른 이유는 단순히 "크기가 작아서"가 아닙니다. **"공유하는 자원이 많기 때문에 문맥 교환 시 버려야 할 하드웨어 캐시(TLB, CPU Cache) 정보를 보존할 수 있고, 주소 공간을 재설정하는 무거운 시스템 작업이 생략되기 때문"**입니다. 

따라서 동시성 프로그래밍을 설계할 때, 독립성이 절대적으로 중요한 작업이 아니라면 오버헤드가 적은 멀티쓰레딩이 멀티프로세싱보다 성능 측면에서 압도적으로 유리합니다.
